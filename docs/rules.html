Projeto Darkpath

Darkpath é um jogo demo sendo desenvolvido por estudantes do curso de iGames da Infinite Loop.

GUIDELINES DO PROJETO

  1. REPOSITÓRIO - Todos os programadores devem procurar ter acesso ao repositóro Github, passando seu email e/ou nickname para o lider da programação.

    Todos os programadores devem ter uma conta no Github e ter a versão da Unreal Engine igual a do projeto.

    Todos os programadores precisam baixar o Github for Desktop, para sincronizar o projeto com o repositório, enviar modificações e receber modificações.

    Na primeira vez que, você, programador sincronizar o projeto com seu PC, o launcher da Unreal Engine deve automaticamente detectar o projeto. Caso isso não aconteça, procure na sua pasta Documentos, e execute o arquivo .UE4Project.

    Toda vez que for fazer uma modificação, abra o Github for Desktop e de um Pull Changes para receber todas as modificações recentes. Caso tenha terminado de modificar, não esqueça de fazer um Commit, detalhando o mais possivel suas modificações para facilitar o versionamento e uma atualização ao contrário; e depois faça um Push Changes, para que todos recebam suas modificações.

    Quando encontrar algum bug no projeto, o colaborador ou tester deve imediatamente ir no repositório e criar um Issue para que os programadores saibam que existe um bug.

    Quando o programador lider ou o lider de projeto ver o Issue criado, irá imediatamente classificar ele e alertar os programadores.

    Exemplo: Issue #1

    Quando um Issue for consertado, na parte de Correções no versionamento vai aumentar uma unidade, dependendo de quantos erros consertados.

    E no final, o site será atualizado. 

  2. PROGRAMAÇÃO -  Para mudanças serem feitas na programação do projeto, todas as regras regidas no Item 1 - Repositório devem ser seguidas.

    Todos os programadores que irá usar C ++ precisa instalar os sequintes compliadores, Visual Studio no Windows e Xcode no Mac OS.

    Todo o código envolvido nesse projeto é de propriedade da equipe, contanto, o código poderá ser usado em qualquer outro projeto independente da presença do criador do código, de acordo com a Licença MIT, o dono do código vai receber os créditos sob o código, mas não poderá ganhar em cima deste código.

    Todos os programadores devem seguir a terminologia, para que o projeto tenha um organização decente.

  3. TERMINOLOGIA - Todos os programadores devem seguir uma seguinte terminologia para organizar o projeto, com vista de uma melhor entendimento e compilação.

    A palavra "mapa" geralmente se refere ao que a pessoa média chama de "nível" e pode ser usada de forma intercambiável.

    Existem algumas maneiras diferentes de nomear coisas. Todos devem usar a mesma maneira, que no momento é a Snake_Case:
    As palavras pode começar arbritariamente com maiusculas ou minusculas mas as palavras são separadas usando um Underline. Exemplo: Snake_case, deagle_Weapon.

    Para mais informações sobre a Terminologia acesse: Guide to Terminology (em EN-US)

  4. MODELAGEM 3D - Absolutamente TODOS os arquivos deverão ser salvos/exportados em ‘.obj’, durante o processo e entregues, sem exceções.

    TODOS os arquivos deverão ser entregues para ‘Walter Pimentel’, para serem devidamente revisados e preparados para a engine e/ou animação. 

  5.   TEXTURIZAÇÃO - TODOS os arquivos, antes de finalizados, deverão ser passados para ‘Walter Pimentel’ em .psd ou em .spp (substance painter) para serem devidamente revisados e preparados para a engine. 

    Absolutamente TODOS os arquivos finais deverão ser salvos/exportados em ‘.png’

  6. MAIS INFORMAÇÕES SOBRE ESTRUTURA E TERMINOLOGIA - Acesse o documento oficial criado pelo lider do projeto no Google Drive: Regras de Produção - Walter Pimentel
